
## HailoInfer Multi-Model Refactor ŌĆö 2026-05-11

### Background

The original `HailoInfer` was designed exclusively for single-model inference.
Refactored to support multi-model scenarios such as CLIP zero-shot classification,
where a text encoder and an image encoder must operate under the same device.

---

### Changed Files

#### `src/common/hailo_infer.hpp`

- Added `HailoModelConfig` struct
  - Specifies `hef_path`, `input_type`, and `output_type` per model individually
- Added `HailoModel` struct
  - Encapsulates per-model state (`network_group`, `input_vstreams`, `output_vstreams`, `output_vstream_info_by_name`) that was previously spread across class members
- Updated `HailoInfer` class
  - Replaced single `network_group` member with `vector<HailoModel> models`
  - Added multi-model constructor `(vector<HailoModelConfig>)`
  - Existing single-model constructor `(string, format, format)` retained as a delegating constructor for backward compatibility
  - Added `model_index` parameter (default 0) to `get_input_info()`, `get_output_infos()`, and `infer()`
  - Changed `infer()` input type from `cv::Mat` to `hailort::MemoryView` ŌĆö type-agnostic for images, float embeddings, token arrays, etc.
  - Updated `set_input_buffers()` signature accordingly

#### `src/common/hailo_infer.cpp`

- Extracted VDevice creation into a `create_vdevice()` helper
- Constructor now iterates over `configs`, configures each HEF, constructs a `HailoModel`, and pushes it into `models`
- `init_vstreams()`, `set_input_buffers()`, and `infer()` all updated to take `HailoModel&` instead of class-level members
- Replaced `cv::Mat`-specific fields in `infer()` log with `model_index` and `input_size`

#### `src/classification/classifier.cpp`

- Updated `infer()` call site:
  `model.infer(preprocessed, output_buffers)`
  ŌåÆ `model.infer(MemoryView(preprocessed.data, ...), output_buffers)`

#### `src/common/INFER_ARCHITECTURE.md`

- Added `## Multi-Model VStream Inference` section covering:
  - Diagram of single VDevice + multiple `HailoModel` instances
  - Rationale for sharing one VDevice
  - Constructor usage examples with `HailoModelConfig`
  - `MemoryView` input abstraction explanation
  - Per-model `infer()` flow
  - Scheduler behavior comparison: sequential (current) vs concurrent (future)

---

### Design Decisions

| Decision | Rationale |
|---|---|
| Single shared VDevice | Avoids redundant chip initialization; keeps all models under one scheduler context |
| `HailoModelConfig` struct | Each model may require different format types ŌĆö per-model specification is necessary |
| `vector<HailoModel>` | Models are fixed at startup with no runtime additions ŌĆö accepting a list is the simplest approach |
| `MemoryView` input | Provides a uniform interface regardless of input type (pixel data, float embeddings, token arrays) |
| Backward-compatible single-model constructor | Minimizes changes to existing callers such as `classifier.cpp` |
| Sequential execution retained | CLIP runs text encoder once and image encoder per frame ŌĆö concurrent execution offers no benefit |

## HailoInfer Struct Redesign ŌĆö 2026-05-11 (follow-up)

### Background

During multi-model refactoring, `HailoModel` held both caller-facing config fields and
move-only VStream handles in the same struct. This caused a compile error when passing
`vector<HailoModel>` as a constructor argument ŌĆö the compiler attempted to generate a
copy constructor for `HailoModel`, which failed because `InputVStream`/`OutputVStream`
have their copy constructors deleted (move-only HW resource handles).

---

### Root Cause

```
vector<HailoModel> as constructor arg
    ŌåÆ compiler generates HailoModel copy constructor
        ŌåÆ attempts to copy InputVStream
            ŌåÆ copy constructor = delete ŌåÆ compile error
```

`const vector<HailoModel>&` does not help ŌĆö copyability is checked at type definition
time, not at the point of actual copy.

---

### Changed Files

#### `src/common/hailo_infer.hpp`

Separated one struct into three with distinct responsibilities:

- `HailoModel` ŌĆö caller-facing config only (`hef_path`, `input_type`, `output_type`)
  - Fully copyable ŌĆö no VStream members
  - Used as constructor argument

- `HailoVStreamPool` ŌĆö HW buffer handles only (`input_vstreams`, `output_vstreams`)
  - Move-only ŌĆö owns DMA buffer handles
  - Never passed across API boundary

- `HailoModelState` ŌĆö internal per-model runtime state
  - Contains `HailoModel config`, `HailoVStreamPool pool`, `network_group`, `output_vstream_info_by_name`
  - Stored in `vector<HailoModelState>` inside `HailoInfer` via `std::move`

- `init_vstreams()` signature simplified ŌĆö reads `input_type`/`output_type` from `state.config` directly, no longer takes them as separate parameters

#### `src/common/hailo_infer.cpp`

- `create_vdevice()` helper removed ŌĆö inlined back into constructor (was used only once)
- All internal references updated from flat `model.*` to `state.config.*` and `state.pool.*`
- `infer()` references pool via `auto &pool = state.pool` for clarity

#### `src/classification/classifier.cpp`

- Buffer allocation uses `get_output_type()` with three-way elem_size:
  `FLOAT32 ŌåÆ sizeof(float)` / `UINT16 ŌåÆ sizeof(uint16_t)` / default `ŌåÆ sizeof(uint8_t)`

---

### Design Decisions

| Decision | Rationale |
|---|---|
| Split `HailoModel` / `HailoVStreamPool` / `HailoModelState` | VStream is move-only ŌĆö config and HW handles cannot coexist in a copyable struct |
| `HailoModel` as constructor arg | Copyable config struct crosses the API boundary cleanly |
| `HailoVStreamPool` never exposed publicly | HW handles are an internal concern ŌĆö callers have no reason to access them directly |
| `state.config` / `state.pool` sub-struct access | Makes ownership and responsibility explicit at each call site |
