
## Config Unification & Project Cleanup — 2026-05-12

### Background

Task-specific config files were scattered across each task directory.
Unified into a single root `config.yaml` with a `task` key for switching,
and added `starter.sh` to build and run the selected task in one command.

---

### Changed Files

#### `config.yaml` (root, newly created)

- Single config for all tasks
- Root-level common fields: `task`, `input`, `output`
- Task-specific blocks: `classification`, `zero_shot_classification`
- `encoder_path` / `npy_path` prefix keys — filenames specified separately under each block
- `prompts_file` key — references `text_label.yaml` instead of inlining prompts

#### `src/common/cli.hpp`

- Introduced `BaseConfig` struct — holds `task`, `input`, `output`
- `Config` and `ZeroShotConfig` inherit from `BaseConfig`
- `load_base_config(cfg, node)` — parses common fields from root yaml node
- `add_base_options(app, cfg)` — registers common CLI options
- `load_config` / `load_zeroshot_config` now read from root yaml and index into task-specific sub-nodes
- `join_path()` helper — concatenates `encoder_path`/`npy_path` prefix with filename
- `prompts_file` support — loads prompts from an external yaml file; inline `prompts` still appended if present

#### `src/zero_shot_classification/clip_example.cpp`

- Config path changed from `"src/zero_shot_classification/config.yaml"` → `"config.yaml"`

#### `src/classification/classifier.cpp`

- Config path changed from `"src/classification/config.yaml"` → `"config.yaml"`

#### `starter.sh` (newly created)

- Reads `task` from root `config.yaml` via `grep`
- Resolves task directory and binary path based on task name
- Runs cmake build then executes the binary from `ROOT_DIR`
- Passes `LOG_LEVEL` env var and extra CLI args (`$@`) through to the binary

#### `src/zero_shot_classification/text_label.yaml`

- 100+ prompts across animals, vehicles, electronics, buildings, food, etc.
- Referenced via `prompts_file` in `config.yaml` — not inlined

---

### Design Decisions

| Decision | Rationale |
|---|---|
| Single root `config.yaml` | One place to change task, input, output — no need to touch task-specific files |
| `BaseConfig` inheritance | Common fields defined once; new tasks inherit without duplication |
| `prompts_file` indirection | Prompt list can grow independently without cluttering `config.yaml` |
| `starter.sh` reads `task` via grep | Avoids a yaml parser dependency in the shell script |
| Binaries run from `ROOT_DIR` | Relative paths in `config.yaml` (e.g. `local_data/`) resolve correctly |

## HailoVStreamPool Removal — 2026-05-12

### Background

`HailoVStreamPool` was introduced during the struct redesign to group
`input_vstreams` and `output_vstreams`. On review, it was only ever used
inside `HailoModelState` and was never passed independently — an unnecessary
layer of nesting with no semantic value.

---

### Changed Files

#### `src/common/hailo_infer.hpp`

- Removed `HailoVStreamPool` struct entirely
- `input_vstreams` and `output_vstreams` moved directly into `HailoModelState`

#### `src/common/hailo_infer.cpp`

- All `state.pool.input_vstreams` / `state.pool.output_vstreams` references flattened to `state.input_vstreams` / `state.output_vstreams`
- `auto &pool = state.pool` local alias in `infer()` removed; uses `state` directly

---

### Design Decisions

| Decision | Rationale |
|---|---|
| Remove `HailoVStreamPool` | Used only inside `HailoModelState`, never passed independently — nesting added complexity without benefit |
| Flatten into `HailoModelState` | `state.input_vstreams` is one level shallower and equally clear |

## HailoInfer Multi-Model Refactor — 2026-05-12

### Background

The original `HailoInfer` was designed exclusively for single-model inference.
Refactored to support multi-model scenarios such as CLIP zero-shot classification,
where a text encoder and an image encoder must operate under the same device.

---

### Changed Files

#### `src/common/hailo_infer.hpp`

- Added `HailoModelConfig` struct
  - Specifies `hef_path`, `input_type`, and `output_type` per model individually
- Added `HailoModel` struct
  - Encapsulates per-model state (`network_group`, `input_vstreams`, `output_vstreams`, `output_vstream_info_by_name`) that was previously spread across class members
- Updated `HailoInfer` class
  - Replaced single `network_group` member with `vector<HailoModel> models`
  - Added multi-model constructor `(vector<HailoModelConfig>)`
  - Existing single-model constructor `(string, format, format)` retained as a delegating constructor for backward compatibility
  - Added `model_index` parameter (default 0) to `get_input_info()`, `get_output_infos()`, and `infer()`
  - Changed `infer()` input type from `cv::Mat` to `hailort::MemoryView` — type-agnostic for images, float embeddings, token arrays, etc.
  - Updated `set_input_buffers()` signature accordingly

#### `src/common/hailo_infer.cpp`

- Extracted VDevice creation into a `create_vdevice()` helper
- Constructor now iterates over `configs`, configures each HEF, constructs a `HailoModel`, and pushes it into `models`
- `init_vstreams()`, `set_input_buffers()`, and `infer()` all updated to take `HailoModel&` instead of class-level members
- Replaced `cv::Mat`-specific fields in `infer()` log with `model_index` and `input_size`

#### `src/classification/classifier.cpp`

- Updated `infer()` call site:
  `model.infer(preprocessed, output_buffers)`
  → `model.infer(MemoryView(preprocessed.data, ...), output_buffers)`

#### `src/common/INFER_ARCHITECTURE.md`

- Added `## Multi-Model VStream Inference` section covering:
  - Diagram of single VDevice + multiple `HailoModel` instances
  - Rationale for sharing one VDevice
  - Constructor usage examples with `HailoModelConfig`
  - `MemoryView` input abstraction explanation
  - Per-model `infer()` flow
  - Scheduler behavior comparison: sequential (current) vs concurrent (future)

---

### Design Decisions

| Decision | Rationale |
|---|---|
| Single shared VDevice | Avoids redundant chip initialization; keeps all models under one scheduler context |
| `HailoModelConfig` struct | Each model may require different format types — per-model specification is necessary |
| `vector<HailoModel>` | Models are fixed at startup with no runtime additions — accepting a list is the simplest approach |
| `MemoryView` input | Provides a uniform interface regardless of input type (pixel data, float embeddings, token arrays) |
| Backward-compatible single-model constructor | Minimizes changes to existing callers such as `classifier.cpp` |
| Sequential execution retained | CLIP runs text encoder once and image encoder per frame — concurrent execution offers no benefit |

## HailoInfer Struct Redesign — 2026-05-11 (follow-up)

### Background

During multi-model refactoring, `HailoModel` held both caller-facing config fields and
move-only VStream handles in the same struct. This caused a compile error when passing
`vector<HailoModel>` as a constructor argument — the compiler attempted to generate a
copy constructor for `HailoModel`, which failed because `InputVStream`/`OutputVStream`
have their copy constructors deleted (move-only HW resource handles).

---

### Root Cause

```
vector<HailoModel> as constructor arg
    → compiler generates HailoModel copy constructor
        → attempts to copy InputVStream
            → copy constructor = delete → compile error
```

`const vector<HailoModel>&` does not help — copyability is checked at type definition
time, not at the point of actual copy.

---

### Changed Files

#### `src/common/hailo_infer.hpp`

Separated one struct into three with distinct responsibilities:

- `HailoModel` — caller-facing config only (`hef_path`, `input_type`, `output_type`)
  - Fully copyable — no VStream members
  - Used as constructor argument

- `HailoVStreamPool` — HW buffer handles only (`input_vstreams`, `output_vstreams`)
  - Move-only — owns DMA buffer handles
  - Never passed across API boundary

- `HailoModelState` — internal per-model runtime state
  - Contains `HailoModel config`, `HailoVStreamPool pool`, `network_group`, `output_vstream_info_by_name`
  - Stored in `vector<HailoModelState>` inside `HailoInfer` via `std::move`

- `init_vstreams()` signature simplified — reads `input_type`/`output_type` from `state.config` directly, no longer takes them as separate parameters

#### `src/common/hailo_infer.cpp`

- `create_vdevice()` helper removed — inlined back into constructor (was used only once)
- All internal references updated from flat `model.*` to `state.config.*` and `state.pool.*`
- `infer()` references pool via `auto &pool = state.pool` for clarity

#### `src/classification/classifier.cpp`

- Buffer allocation uses `get_output_type()` with three-way elem_size:
  `FLOAT32 → sizeof(float)` / `UINT16 → sizeof(uint16_t)` / default `→ sizeof(uint8_t)`

---

### Design Decisions

| Decision | Rationale |
|---|---|
| Split `HailoModel` / `HailoVStreamPool` / `HailoModelState` | VStream is move-only — config and HW handles cannot coexist in a copyable struct |
| `HailoModel` as constructor arg | Copyable config struct crosses the API boundary cleanly |
| `HailoVStreamPool` never exposed publicly | HW handles are an internal concern — callers have no reason to access them directly |
| `state.config` / `state.pool` sub-struct access | Makes ownership and responsibility explicit at each call site |
